# AGENTS.md

## Project Overview

膨大なPDF文書にマークづけをしていくことで、それらの関連性を閲覧できるアプリ。
関連性をつけることで文書内の様々な情報の関係や更新による変化、事前に設定した条件との整合性を瞬時に把握できる。

## Directory Structure

編集対象の各子プロジェクトディレクトリの直下に`AGENTS.md`がある場合、それらの内容も読み込んでからタスクに取り組む。

| ディレクトリ       | 役割                                                                 |
| ------------------ | -------------------------------------------------------------------- |
| `docs/`            | プロジェクトの設計資料（`01_要件定義.md`は必ず読みこと。）           |
| `src/`             | フロントエンド + 統合バックエンド (Vue + Typescript + SCSS, PWA統合) |
| `src/components/`  | フロントエンドの画面要素を実装するコンポーネント群                   |
| `src/services/`    | ビジネスロジックを記述するサービス層                                 |
| `src/repositorys/` | LocalStorageをはじめとする各種外部システムとの通信を担うリポジトリ層 |
| `src/models/`      | Zodで定義したスキーマや型定義を格納する                              |
| `output/`          | AI出力ファイルの保存フォルダ                                         |

## Commands

- Add dependency: `bun add hogehoge1 hogehoge2`
- Add dependency (only dev): `bun add -D hogehoge1 hogehoge2`
- test: `bun run test`
- Frontend (with integrated backend): `bun run dev`
- build: `bun run build`

## Code Style

- ファイル名: `sampleProgramFile.ts` 形式
- コメント：関数の冒頭には必ずdocstring形式で説明文を入れる。コード中には処理の流れが追える程度に入れる。
- ファイル分割：機能別、役割別に適切に行う。またファイル構造も設計思想に準じて適切に分割する。
- コメントに用いる言語: 日本語
- API呼び出し：
  - フロントエンド： `api = useBackendApi()`で呼び出す（すべてのバックエンド機能を統合）
  - バックエンド：repository層が各DBや解析器に処理をAPI経由で投げる
- バックエンドの型管理：`Zod`を活用する。型定義は `src/model/schemas.ts` で一元管理（細かい内訳の定義が続く場合はフォルダ分けして総覧性を高める）
- データ永続化：
  - LocalStorage: DB接続情報、初期化フラグ、キャッシュメタデータ
  - IndexedDB: (サイズの大きなデータのキャッシュ。今後深度化。)

## Boundaries

- `.env*` ファイルを変更・コミットしない
- 速やかにデモを完成させるために外部からデータが必要な場合はモックを作成する。外部データが存在しないにもかかわらず勝手に存在するかのように指定しない。
- 作り終えてから随時確認するため、頻繁に方針を確認する必要はない。不明箇所は一度対応策を考えて実装し、その方針を終了時に提案してもらえればよい。
- 実装したコードはコマンドに従って実行し、エラーが生じる場合には修正を行う。修正せずにタスクを完了してはいけない。
- フロントエンド実装の際に`vue`ファイルにそのまま画面に表示するテキストを記載しない。表示テキストはすべて`i18n`を通して表示させる。
- フロントエンドの実装時にコンポーネント分割を意識する。共通化できる箇所はコンポーネントで実装してそれらを呼び出すようにし、`src/pages`の実装は画面の大きな構成がわかりやすいように実装する。
